# 流水线全景

## 定义

开发流水线（Development Pipeline）是一组 12 步的 AI 辅助开发框架，每一步针对一类特定错误。它通过三层架构——方法框架（Method）、行为约束（Behavioral）、流程闸门（Process Gates）——组织所有步骤和检查点，确保每一步都有独立的验证方式。

```
收到任务
  │
  ├─ ① 判定风险层级（→ core/03-risk-tiers.md）
  │    Low / Mid / High — 决定执行哪些步骤
  │
  ├─ ② 执行步骤子集（→ core/02-pipeline-steps.md）
  │    ┌──────┬──────────────────┬──────────────────┐
  │    │  0   │ 工作区模式判定    │ workspace-audit   │
  │    │  1   │ 需求探索          │ brainstorming     │
  │    │  2   │ 方案规划          │ writing-plans     │
  │    │  3   │ 架构审查          │ plan-eng-review   │
  │    │    4 │ 工作树隔离        │ git-worktrees     │
  │    │  5   │ 测试驱动开发      │ TDD               │
  │    │  6   │ 编写代码          │ Karpathy 约束     │
  │    │  7   │ 代码审查          │ review            │
  │    │  8   │ 代码简化          │ simplify          │
  │    │  9   │ 自动化 SAST       │ semgrep           │
  │    │ 10   │ 验证确认          │ verification      │
  │    │ 11-12│ 交付 + 分支收尾   │ ship + cleanup    │
  │    └──────┴──────────────────┴──────────────────┘
  │
  ├─ ③ 双代理互审 Gate ①+②（→ core/04-dual-agent-audit.md）
  │    高/中影响 → 外部审计 agent 审查 → 收敛 → 放行/阻塞
  │
  └─ ④ 交付

三层架构（→ core/05-three-layer-architecture.md）：
┌─ Layer 3: Process Gates ────────────────────────────┐
│  流程闸门 — 决定何时推进、何时停止                     │
│  架构审查 ▸ Gate ① ▸ 代码审查 ▸ simplify ▸ SAST ▸ Gate ② ▸ ship │
├─ Layer 2: Behavioral Constraints ───────────────────┤
│  行为约束 — 始终激活的编码规范                          │
│  先思考 → 最小化 → 精准变更 → 可验证                   │
├─ Layer 1: Method Framework ─────────────────────────┤
│  方法框架 — 如何做事情                                 │
│  探索 → 规划 → TDD → 编码 → 验证                       │
└─────────────────────────────────────────────────────┘
```

**核心原则：错误率 = 流水线的漏洞数。** 每一步只防一类错误。跳一步，该类错误就没有任何防范机制。流水线不提供捷径——捷径等于未覆盖的错误面。

## 适用边界

**在范围内：**

- 非平凡开发任务的流水线整体规划和决策
- 理解各组件（步骤定义、风险判定、双代理互审、三层架构）之间的关系
- 选择阅读路径（根据当前需求决定从哪个组件切入深入）
- 流水线的哲学基础和设计原则

**不在范围内：**

- 步骤的详解和执行细则 —— 详见 `02-pipeline-steps.md`
- 风险层级的具体判定标准 —— 详见 `03-risk-tiers.md`
- 双代理互审的触发规则和执行细节 —— 详见 `04-dual-agent-audit.md`
- 三层架构的层间交互规则 —— 详见 `05-three-layer-architecture.md`

## 正文

### 决策流

流水线的执行不是一个线性"从头走到尾"的过程，而是一个多分支决策：

```
收到任务 ──→ 判定风险层级 ──→ 选择步骤子集 ──→ 执行步骤 ──→ 交付
  ↑                │               │                │
  │                ▼               ▼                ▼
  │         core/03-         core/02-          core/04-
  │         risk-tiers.md    pipeline-         dual-agent-
  │                          steps.md          audit.md
  │                                              │
  └──── scope 变化后重新判定 ────────────── 外部审计收敛 ──
```

1. **判定风险层级**（`03-risk-tiers.md`）：任务按能力域定级 Low / Mid / High。治理规则优先——任何削弱闸门的改动至少升一级。Scope 变化后必须重新判定。
2. **选择步骤子集**（`02-pipeline-steps.md`）：Low 层级跳过架构审查、工作树、TDD、代码审查、简化、交付、分支收尾；Mid/High 执行全部 12 步；High 额外增加外部审计硬闸门。
3. **执行步骤**（`02-pipeline-steps.md`）：逐步骤执行，每步有明确的设计原理（Why This Step Exists）和常见陷阱（常见错误 + 红旗）。
4. **双代理互审**（`04-dual-agent-audit.md`）：Gate ①（架构审计）和 Gate ②（交付审计）在关键节点阻止有质量缺陷的产物向下游传递。高/中影响变更在 Gate ② 经过外部审计 agent 审查，R1-R4 收敛协议确保问题有闭环。

### 与外部组件的关系

流水线不是孤立的——它与两侧的组件协作形成完整的 AI 辅助开发体系：

- **工具映射**（`adapters/tool-mapping-table.md`）：将每步的能力名映射到具体参考工具和替代方案。
- **审计 agent 适配**（`adapters/codex-auditor.md` - 2026-06-07更新）：Codex 作为外部审计 agent 的接口定义、审计发起/响应的 JSON schema、收敛流程和替代 agent 的最小接口要求。
- **Claude Code 适配**（`adapters/claude-code.md`）：Claude Code CLI 环境下各工具的安装和配置指南。
- **配置示例**（`examples/` 目录）：可直接复制使用的 skill 文件、CLAUDE.md、settings.json 骨架。
- **附录**（`appendix/` 目录）：文件触发样例（非规范性）、真实案例复盘。

### 流水线为什么需要这些组件

- **12 步不是发明出来的，是从错误中反推出来的。** 每一类重复出现的错误都指向流水线的一个缺失步骤。如果某种错误模式反复发生，说明流水线少了一个步骤。
- **双代理互审不是可选项。** 执行者"不知道自己犯错"——单一 agent 不可能发现自己的系统性盲区。一个不同模型提供商、不同上下文的独立审计 agent 才有机会发现执行 agent 想不到的问题。
- **风险层级不是官僚主义。** 不是每项任务都需要 12 步全执行。Low 层级跳过非必要步骤，Mid/High 增加严格度——层级选择是一个成本效益决策，不是规则的死板应用。

## 常见错误

| 错误 | 现实 |
|------|------|
| **跳过判定直接开工** —— "这个很简单，不用定级" | 所谓"简单"的任务往往有隐藏风险。不做风险判定就跳过步骤，等于在不知道漏洞的情况下决定不填补。至少用 30 秒跑一遍判定流程。 |
| **以为流水线只有步骤** | 流水线 = 步骤 + 风险判定 + 双代理互审 + 三层架构。只执行步骤列表而不做风险判定和外部审计，相当于有了枪但没装准星。 |
| **开始读 core/01 但不知道目标方向** | 本文件是导航图，不是操作手册。想操作 → 直接去 `02-pipeline-steps.md`。想理解为什么 → 去 `05-three-layer-architecture.md`。在全景图上死磕不会帮到你。 |
| **看了全景就觉得懂了全部 12 步的原理** | 全景图只能告诉你"有这些步骤"，不能替代 `02-pipeline-steps.md` 中每步的 Problem、Solution、Catches。每一个"为什么这步存在"都来自真实开发中的失败教训，不是理论推导。 |
| **把组件间的引用当装饰** | 本文档中的每处引用（→02、→03、→04、→05）都是"这条信息不在这里，去那里看"的语义指针。忽略引用意味着只读到本体半的信息。 |
