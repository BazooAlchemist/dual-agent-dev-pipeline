# 三层架构

## 定义

三层架构将 AI 辅助开发流水线的全部机制分为三个层次——方法框架（Method）、行为约束（Behavioral）、流程闸门（Process Gates）。每层覆盖一类开发风险，层间通过明确的交互规则协同。理解这三层，就理解了流水线"为什么这样设计"。

## 适用边界

**范围内**：
- 解释流水线的三个逻辑层次及其职责分工
- 定义层间交互规则（哪层在何时影响哪层）
- 将流水线的全部步骤归入三层架构

**范围外**：
- 不包含每步详解（见 `02-pipeline-steps.md`）
- 不包含风险层级定义（见 `03-risk-tiers.md`）
- 不包含双代理互审细节（见 `04-dual-agent-audit.md`）
- 不涉及特定工具安装或配置

## 三层的职责边界

```
┌─ Layer 3: Process Gates ────────────────────────────┐
│  流程闸门 — 决定何时推进、何时停止                      │
│  能力：架构审查（plan-eng-review）→ [Gate ①]           │
│        → 代码审查（review）→ 简化（simplify）            │
│        → SAST 扫描（semgrep）→ [Gate ②] → 交付（ship）  │
│  双代理互审：影响分级 + 双 Gate + 收敛协议               │
├─ Layer 2: Behavioral Constraints ───────────────────┤
│  行为约束 — 始终激活的编码规范                           │
│  先思考 → 最小化 → 精准变更 → 可验证                    │
├─ Layer 1: Method Framework ─────────────────────────┤
│  方法框架 — 如何做事情                                  │
│  探索（brainstorming）→ 规划（writing-plans）→           │
│  测试驱动开发（test-driven-development）→                │
│  验证（verification-before-completion）                 │
└─────────────────────────────────────────────────────┘
```

### Layer 1: 方法框架（Method）

方法框架是流水线的基础层，定义了在 AI 辅助下"如何做事情"。它提供一套结构化的步骤序列，覆盖从需求理解到交付验证的全过程。

**职责**：
- 提供完整的开发步骤骨架，确保每一步有明确的输入、输出和验证标准
- 覆盖从需求理解（brainstorming）、方案规划（writing-plans）、工作区隔离（worktrees）、测试驱动开发（test-driven-development）、编码（Karpathy 约束激活）到完成验证（verification-before-completion）的全过程
- 为行为约束层和流程闸门层提供执行上下文

**归属步骤**：Step 0（工作区模式判定）、Step 1（需求探索）、Step 2（方案规划）、Step 4（工作区隔离）、Step 5（测试驱动开发）、Step 6（编码）、Step 10（完成验证）、Step 12（分支清理）、Handoff（上下文移交）

### Layer 2: 行为约束（Behavioral Constraints）

行为约束是始终激活的编码规范层，在编码全过程中持续约束 AI 的行为倾向。它源自"先思考、最小化、精准、可验证"四条准则，无论使用什么 AI 工具都适用。

**四条约束**：

1. **先思考（Think first）**：编码前先理解问题，不直接用 AI 生成代码作为探索手段。调用分析工具（codegraph_context、WebSearch）获取上下文后再编码。
2. **最小化（Minimal）**：用最少的代码解决问题，不引入假设性的未来需求。不添加非必需的抽象层。
3. **精准变更（Surgical）**：只修改需要改动的部分，不动无关代码。不为了风格一致性而重构未改动的模块。
4. **可验证（Verifiable）**：每个变更必须有对应的验证方法。不能验证的变更不应提交。

**职责**：
- 在编码过程中持续约束 AI 的过度设计倾向
- 作为方法框架层的运行时护栏，不接收 PASS/FAIL 判定
- 与流程闸门层不同：行为约束是持续状态，不是阶段检查

### Layer 3: 流程闸门（Process Gates）

流程闸门是流水线的最外层，在关键节点设置检查点，决定是否推进或停止。

**关键闸门**：

- **架构审查（plan-eng-review）**：锁定架构后、写代码前。这是最后一个低成本的变更窗口。
- **Gate ①（架构审计）**：高影响变更触发的外部架构审计。审计对象：架构审查输出 + 实现计划。
- **代码审查（review）**：基于 diff 的正确性检查，寻找"CI 绿色但生产环境故障"类的 bug。
- **简化审查（simplify）**：审查后的代码质量优化，不替代代码审查。
- **SAST 扫描（semgrep）**：自动化模式漏洞检测（注入、硬编码凭据等）。
- **Gate ②（交付审计）**：高/中影响变更触发的外部交付审计。审计对象：全部审查产出 + 验证结果。
- **交付自动化（ship）**：版本号、变更日志、合并、PR。

**职责**：
- 在关键决策点设置强制检查，确保架构、代码、安全都经过独立验证
- 管理外部审计 agent 的介入时机和收敛流程
- 风险层级判定（见 `03-risk-tiers.md`）决定哪些闸门激活

## 层间交互规则

三层不是层级分明的控制系统——每层有不同的作用时机和条件。

1. **上层不越俎代庖**：流程闸门决定"要不要推进"，但不告诉方法框架"怎么做"。Gate ① 可以驳回不完善的架构计划，但不能指定开发者用什么步骤来改进。

2. **下层向上层报告状态**：方法框架中的步骤（如测试失败）可以触发流程闸门层的拦截（如"阻断交付直到测试通过"）。行为约束层的持续违规（如过度设计）可以通过代码审查（review）在流程闸门层被检出。

3. **行为约束始终激活**：Layer 2 覆盖 Layer 1 的编码步骤全生命周期。无论编码处于哪个阶段，四条约束始终生效。行为约束不是一层"开关"——它是始终运行的后台规则。

4. **流程闸门只拦截不替代**：Gate ① 和 Gate ② 的驳回意味着"当前产出不满足交付标准"，返回的是审查发现和修复指引，不是替代方案。执行者自行确定如何修复。

5. **同层内互相引用**：方法框架层的步骤详解（`02-pipeline-steps.md`）引用同层的风险层级（`03-risk-tiers.md`）来选择步骤子集。流程闸门层的审核协议（`04-dual-agent-audit.md`）引用同层的闸门流程来定义触发规则。

## 常见错误

| 错误 | 正确理解 |
|------|---------|
| **把层当装饰**：三层写进文档，但实际开发跳过了所有闸门 | 每层对应一类开发风险。漏一层 = 漏一类错误。三层不是"建议阅读"的装饰内容 |
| **跳层执行**：不经方法框架探索直接进入编码，或不经流程闸门直接交付 | 层是串行的，不是可选并行的。跳过探索直接编码，等于放弃了对方向的验证 |
| **反向依赖**：让方法框架层决定流程闸门的触发规则（如"代码写得完美就不需要外部审计"） | 流程闸门的触发逻辑由风险层级（`03-risk-tiers.md`）和能力域决定，不由当前产出的主观质量决定 |
| **把行为约束当流程闸门**："先思考"不是一个可以拦截的步骤，是一个持续状态 | 行为约束始终激活，不接收 PASS/FAIL 判定。不要用 Process Gates 的执行方式套用到 Behavioral Constraints |
| **忽视层间交互**：认为三层是独立模块，从不考虑层间影响 | 跨层缺陷是最危险的（见 `appendix/real-world-cases.md` 子案例 2）。每层看起来"有人管"，但全链路无人覆盖 |
