# 风险层级：Low / Mid / High 三级判定体系

## 定义

风险层级是开发流水线根据任务影响范围判定执行严谨度的分级机制。每级对应一组流水线步骤子集——层级越低，跳过的非必要步骤越多；层级越高，执行的步骤越完整，外部审计的门槛越严格。

核心原则：**跳过流水线步骤 = 漏掉该类错误。** 风险层级决定了哪些步骤可以安全跳过，哪些必须执行。

## 适用边界

**在范围内：**

- 新任务的风险定级（开始编码前判定）
- 已有任务的重新定级（scope 变化后重新评估）
- 流水线步骤选择（根据层级确定执行哪些步骤）

**不在范围内：**

- 能力域的具体触发规则（哪些能力域属于高/中/低影响）—— 详见 `04-dual-agent-audit.md`
- 外部审计 agent 的接入和调用流程 —— 详见 `04-dual-agent-audit.md`
- 单步骤的执行细节和常见陷阱 —— 详见 `02-pipeline-steps.md`

## 正文

### 三级定义表

| 层级 | 影响范围 | 典型场景 | 对应流水线步骤 |
|------|---------|---------|--------------|
| **Low** | 单文件或单一功能，不影响其他模块，回滚代价低 | 脚本修改、配置编辑、一次性修复、格式修正、纯文档记录 | 步骤 0-2, 6, 9-10（跳过架构审查、工作树、TDD、代码审查、简化、交付、分支收尾） |
| **Mid** | 跨文件或跨模块变更，影响非关键路径，回滚代价可接受 | 功能开发、hook 修改、非关键代码改动、新增 skill/workflow、方案执行 | 全部 12 步 |
| **High** | 跨系统或影响关键路径，回滚代价高或不可回滚 | 桥接协议、认证与权限、服务生命周期、审查标准削弱、Agent 选择与路由、会话或锁机制 | 全部 12 步 + 外部审计硬闸门 |

### 判定流程

```
收到任务
  │
  ├─ 1. 能力域检查：改动涉及哪些能力域？
  │     能力域定义见 core/04-dual-agent-audit.md
  │     匹配最高层级（高 > 中 > 低）
  │
  ├─ 2. 治理规则优先检查：
  │     是否削弱现有安全控制、删除闸门、放宽审查标准？
  │     → 是：至少升一级
  │
  └─ 3. 得出风险层级
       │
       ├─ Low  → 执行低影响步骤子集
       ├─ Mid  → 执行全部步骤 + Gate ② 按影响配置
       └─ High → 执行全部步骤 + Gate ① + Gate ② 硬闸门
```

**关键规则：**

1. **最高匹配胜出** —— 如果改动涉及多个能力域，按最高层级处理。例如同时修改了文档（低影响）和桥接协议（高影响），按高影响执行全流水线。
2. **治理规则优先** —— 任何削弱现有安全控制的改动，至少升一级。删除一个闸门 = 高影响。
3. **重新定级** —— 如果任务 scope 在执行过程中变化（如从"改文档"变成"改协议"），重新运行判定流程。旧层级不再有效。

### 三级步骤差异对照表

下表展示每个风险层级实际执行的流水线步骤。`✓` 表示执行，`—` 表示跳过。双代理互审 Gate 的触发规则见 `04-dual-agent-audit.md`。

| 步号 | 能力名 | 参考工具 | Low | Mid | High |
|------|-------|---------|:---:|:---:|:----:|
| 0 | 工作区模式判定 | workspace-startup-audit | ✓ | ✓ | ✓ |
| 1 | 需求探索 | brainstorming | ✓ | ✓ | ✓ |
| 2 | 方案规划 | writing-plans | ✓ | ✓ | ✓ |
| 3 | 架构审查 | plan-eng-review | — | ✓ | ✓ |
| 4 | 工作树隔离 | using-git-worktrees | — | ✓ | ✓ |
| 5 | 测试驱动开发 | test-driven-development | — | ✓ | ✓ |
| 6 | 编写代码 | Karpathy Guidelines | ✓ | ✓ | ✓ |
| 7 | 代码审查 | review | — | ✓ | ✓ |
| 8 | 代码简化 | simplify | — | ✓ | ✓ |
| 9 | 自动化 SAST | semgrep | ✓ | ✓ | ✓ |
| 10 | 验证确认 | verification-before-completion | ✓ | ✓ | ✓ |
| ★ | 双代理互审 Gate ①+② | 外部审计 agent | — | 按影响配置 | 按影响配置 |
| 11 | 交付自动化 | ship | — | ✓ | ✓ |
| 12 | 分支收尾 | finishing-a-development-branch | — | ✓ | ✓ |
| — | 上下文交接 | handoff | — | 按需 | 按需 |

**Low 层级跳过步骤的原因：**

- **步骤 3（架构审查）** —— Low 层级的变更（如配置编辑、一次性脚本）不涉及架构层面的决策，架构审查引入的开销超过其价值。
- **步骤 4（工作树隔离）** —— 单文件或一次性修改不需要分支隔离。如果后续发现需要上下文切换，可以临时创建。
- **步骤 5（TDD）** —— 测试驱动开发对非关键路径的低风险变更不是必要步骤。但并非禁止——如果变更涉及逻辑判断，仍建议写测试。
- **步骤 7（代码审查）** —— 单文件简单变更的 bug 概率低，内置验证（步骤 10）和 SAST 扫描（步骤 9）已覆盖大部分风险。
- **步骤 8（代码简化）** —— 短脚本或配置的简化收益不抵审查开销。
- **步骤 11（交付自动化）** —— 低风险变更直接应用到目标环境即可，不需要完整的交付流水线。
- **步骤 12（分支收尾）** —— 没有创建分支或工作树，自然不需要清理。

## 常见错误

| 错误 | 现实 |
|------|------|
| **低估影响** —— "我只是改了一行配置" | 一行配置可能在认证、权限或服务生命周期域内，属于高影响。按能力域判定，不按代码行数。 |
| **用文件名判定风险** —— "改了 `*.py` 所以是低风险" | 文件类型不决定风险层级。同一个 `.py` 文件可以是工具脚本（低影响），也可以是桥接协议实现（高影响）。 |
| **忽视治理规则优先** —— "加个 bypass 开关，只有我能用" | 任何削弱闸门的改动，无论物理影响多大，至少升一级。"只有我能用"的 bypass 在别人看来就是系统漏洞。 |
| **scope 变了不重新定级** —— "已经做完判定了，继续用原来的层级" | Scope 变化后必须重新运行判定流程。从"改注释"变成"改逻辑"——层级可能从 Low 跳到 Mid 或 High。 |
