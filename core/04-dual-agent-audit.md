# 双代理互审机制

## 定义

双代理互审（Dual-Agent Audit）是开发流水线中通过两个独立 agent 交叉验证、防止"执行者不知道自己犯错"的安全闸门。执行 agent 受限于自身上下文和视角——它不知道自己在哪一步犯了错。一个不同模型提供商、不同上下文、不同视角的独立审计 agent 能够发现执行 agent 的盲区。影响分级（高/中/低）的完整定义见 `03-risk-tiers.md`。

**解决的问题**：单 agent 执行时，有且仅有一个视角判断质量。"我觉得没问题"不等于"真的没问题"。外部审计 agent 与执行 agent 之间没有共享上下文、没有共享推理路径、没有共享盲区——这是它唯一且最大的价值。

## 适用边界

**在范围内**：
- 开发流水线各步骤的交付物需要第二视角确认时
- 架构决策、关键实现、交付安全需要外部审计时
- 不同模型提供商的独立 agent 作为审计方

**不在范围内**：
- 替代单元测试、集成测试或端到端测试
- 替代 lint、类型检查或自动化 SAST（这些由流水线其他步骤覆盖）
- 替代代码审查（code review）或结对编程
- 替代"确保功能正确"的运行时验证

## 影响分级判定

改动按涉及的**能力域**定级，不按文件名。最高匹配层级胜出。治理规则优先——任何削弱安全控制的改动升一级。

**判定原则**：
1. 最高匹配层级胜出（改动涉及多个能力域时，按最高影响定级）。
2. 治理规则优先：任何删除闸门、放宽审查标准、降低影响定级、绕过验证步骤的规则修改，至少升一级处理。

### 高影响 -- 外部审计 agent 为硬闸门

高影响变更必须在交付前经过外部审计 agent 审查。外部审计 agent 不可用时，阻塞等待恢复（或走紧急热修复 break-glass 路线）。

| 能力域 | 说明 |
|--------|------|
| Agent 选择与路由 | 改变使用哪个模型、哪个 provider、哪个 agent 类型处理任务 |
| 桥接协议 | 改变 agent 之间的通信协议、消息格式、任务分发逻辑 |
| 会话/锁机制 | 改变 session 生命周期、并发控制、锁管理 |
| 认证与权限 | 改变 token 管理、权限范围、访问控制 |
| 服务生命周期 | 改变守护进程的启动/停止/健康检查配置 |
| 审查标准削弱 | 任何删除闸门、放宽审查标准、降低影响定级、绕过验证步骤的规则修改 |

### 中影响 -- 外部审计优先，不可用时降级审查

中影响变更优先由外部审计 agent 审查。外部审计 agent 不可用时，走降级审查机制（见下方）。

| 能力域 | 说明 |
|--------|------|
| 业务逻辑处理 | 非认证相关的 hook、handler、数据处理逻辑 |
| 新增 skill/workflow | 扩展自动化能力但不改变既有闸门 |
| 治理规则补充 | 新增规则、补充说明（非削弱性修改） |
| 方案执行 | 将已批准的方案落地为代码 |

### 低影响 -- 内置验证足够，无需外部审计

低影响变更不需要外部审计 agent，流水线内置的验证步骤（单元测试、lint、类型检查等）足够覆盖。

| 能力域 | 说明 |
|--------|------|
| 格式修正 | 缩进、注释、非语义调整 |
| 一次性诊断脚本 | 排查用的临时脚本 |
| 纯文档记录 | 非规则类的 .md 文件 |

> **非规范性附录**：能力触发器对应的文件举例参见 `appendix/file-trigger-examples.md`。注意：文件举例不是判定规则。文件改了但能力域未触及，不定为高影响。

## 双 Gate 流程

双代理互审在流水线中设置两道条件性交付点。Gate ① 守架构决策的合理性，Gate ② 守交付物的安全性。

```
收到改动 → 影响分级判定（按能力域，最高匹配胜出，治理规则优先）
                  │
                  ▼
          ┌── Gate ①（架构审计）──┐
          │  插入位置：架构审查后 │
          │  审查对象：架构设计   │
          │  与实现计划           │
          └─────────┬────────────┘
                    │
         ┌──────────┴──────────┐
         ▼                     ▼
    高影响变更            中/低影响变更
         │                     │
         ▼                     ▼
  外部审计 agent 审查     跳过 Gate ①
     ┌───┴───┐           （架构审查步骤
     │       │             本身已足够）
     ▼       ▼
   通过     不通过
     │       │
     │    修改后重审
     │       │
     ▼       ▼
   ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
          │
          ▼
    实现 + 测试 + 验证
          │
          ▼
   ┌── Gate ②（交付审计）──┐
   │  插入位置：全部验证    │
   │  完成后、交付前        │
   └─────────┬────────────┘
             │
   ┌─────────┴─────────┐
   ▼                   ▼
高影响变更          中影响变更
   │                   │
   ▼                   ▼
外部审计 agent     外部审计 agent
  硬闸门              优先
   ┌───┴───┐       ┌───┴───┐
   ▼       ▼       ▼       ▼
 可用    不可用   可用    不可用
   │       │       │       │
   ▼       ▼       ▼       ▼
 审计     阻塞    审计    降级审查
 通过    (或 break- 通过   + 24h 补审
         glass)
   │               │       │
   ▼               ▼       ▼
   ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
             │
             ▼
        低影响变更 → 无额外闸门
        （流水线内置验证足够）
```

#### Gate ①（架构审计）

- **插入位置**：架构审查步骤之后、写第一行代码之前
- **审查对象**：架构审查输出 + 实现计划
- **高影响**：外部审计 agent 审计架构/设计产物。不可用 → 阻塞等待（或走紧急热修复）
- **中/低影响**：跳过 Gate ①（架构审查步骤本身已为设计意图提供了足够的验证）

#### Gate ②（交付审计）

- **插入位置**：代码审查、简化、自动化 SAST、自我验证全部完成后、交付前
- **高影响**：外部审计 agent 硬闸门。可用 → 审计通过（0 BLOCKER）→ 放行。不可用 → 硬阻塞（或走 break-glass）
- **中影响**：外部审计 agent 优先。可用 → 审计通过 → 放行。不可用 → 降级审查 → 放行 + 事后补审
- **低影响**：无额外闸门，流水线内置验证足够

## 外部审计 agent 可用性判定

"外部审计 agent 可用"必须同时满足以下两项：

1. 外部审计 agent 的桥接进程存活且可通信
2. 当前会话中能够成功发起并收到外部审计响应

不满足任何一项即视为不可用。不可用状态不得由执行 agent 自行判定为"可用"。

## 降级审查机制

仅适用于**中影响 Gate ②**且外部审计 agent 不可用的情况。

条件：
- 变更影响分级为"中影响"
- 外部审计 agent 确认不可用
- 非紧急热修复场景（紧急场景走 break-glass 路径）

步骤：
1. 通过当前执行环境分派一个审查 agent，使用与执行 agent **不同的模型 ID/tier**（不同模型提供商优先）
2. 在审查结论中标注 `DEGRADED REVIEW -- NOT equivalent to dedicated external audit agent`
3. 使用与外部审计对齐的输出 schema（BLOCKER / FIX / NOTE + 最终判定）
4. 降级审查通过（0 BLOCKER）→ 放行
5. 事后 **24 小时内**必须补交标准外部审计。补审 owner 为本次任务执行 agent，必须在执行日志中创建 24 小时截止的外部审计补审任务。交付物（handoff/PR）的 owner 负责跟进关闭。

### 标记规则

| 标记 | 写入位置 | 含义 |
|------|---------|------|
| `[degraded-audit]` | 任务执行日志审查结论区 + 交付物元数据（handoff / PR description） | 降级审查通过，待补审 |
| `[emergency-bypass]` | 同上 + 事后补审任务中引用 | 紧急热修复绕过闸门 |

## 紧急热修复 break-glass

适用于高影响变更 + 外部审计 agent 不可用 + 需立即部署的场景（仅限生产故障、服务中断、安全事件）。

条件（全部满足）：
1. 变更为高影响（或涉及审查标准削弱）
2. 外部审计 agent 确认不可用
3. 正在处理生产故障、服务中断或安全事件
4. 用户通过显式确认授权绕过（如 `!force-merge` 命令或直接审批）

约束：
- 仅限紧急修复场景，不得用于常规功能开发
- 自动标记 `[emergency-bypass]`
- 必须完成最低限度的运行时验证
- 事后 **24 小时内**强制外部审计补审。补审 owner 为本次任务执行 agent
- 补审发现 BLOCKER → 必须立即修复或回滚，不得因为"已部署"而豁免

## R1-R4 收敛协议

外部审计 agent 与执行 agent 之间通过最大 4 轮交互完成收敛。

| 轮次 | 内容 | 判定 |
|------|------|------|
| R1 | 外部审计 agent 初始审查，输出完整审查结果 | BLOCKER / FIX / NOTE |
| R2 | 执行 agent 提交修复方案，外部审计 agent 逐条验证 | RESOLVED / PARTIAL / NOT_FIXED |
| R3 | 追加修复（对 R2 中 PARTIAL/NOT_FIXED 的项目） | 同上 |
| R4 | 最终判定 | PASS / BLOCKED |

- **收敛条件**：0 BLOCKER，所有 FIX 已处理
- **最大轮次**：4 轮。R4 未收敛（判定 BLOCKED）→ 升格到用户决策
- 每轮审查必须记录判定结果
- R4 之后不得自动追加轮次——超过 4 轮意味着问题无法在现有框架内解决，需要人工介入

## 外部审计 agent 接口定义

以下定义外部审计 agent 的最小接口。任何满足此接口的独立 agent 均可作为外部审计方。

### 发起审计

执行 agent 向外部审计 agent 发送审计请求，包含以下字段：

```json
{
  "type": "audit_request",
  "gate": "gate_1|gate_2",
  "version": "1.0",
  "payload": {
    "context": {
      "task": "审计任务描述",
      "risk_level": "high|medium|low",
      "impact_domains": ["受影响的能力域列表"],
      "files_changed": ["变更文件路径列表"]
    },
    "artifacts": {
      "design_doc": "架构设计文档或实现计划（Gate ① 时必填）",
      "diff": "代码变更差异（Gate ② 时必填）",
      "test_results": "测试结果摘要（Gate ② 时可选）",
      "verification_log": "自我验证日志（Gate ② 时可选）"
    }
  }
}
```

### 审计响应

外部审计 agent 返回审查结果：

```json
{
  "type": "audit_response",
  "gate": "gate_1|gate_2",
  "version": "1.0",
  "result": {
    "verdict": "PASS|BLOCKED",
    "findings": [
      {
        "severity": "BLOCKER|FIX|NOTE",
        "category": "architecture|security|correctness|consistency|completeness",
        "location": "问题位置（文件/行）",
        "description": "问题描述",
        "suggestion": "修复建议"
      }
    ],
    "summary": "审查总结"
  }
}
```

- `BLOCKER`：必须修复，否则不得通过
- `FIX`：建议修复，但不阻塞
- `NOTE`：观察项，无需处理

### 收敛（R2-R4）

执行 agent 修复后发送修复响应：

```json
{
  "type": "fix_response",
  "round": 2,
  "fixes": [
    {
      "finding_index": 0,
      "status": "RESOLVED|PARTIAL|NOT_FIXED",
      "action_taken": "修复说明"
    }
  ]
}
```

### 替换为其他 agent 的最小要求

任何独立 agent 替换外部审计角色，必须满足：

1. **模型提供商不同**：与执行 agent 不同 API provider（如执行 agent 使用 Anthropic，审计 agent 使用 OpenAI 或 Google）
2. **上下文隔离**：不共享对话历史、session 状态或中间推理结果
3. **接口兼容**：支持上述 JSON schema（审计请求 + 审计响应 + 收敛轮次）
4. **交付物独立**：审计结果写入独立文件，不影响执行 agent 的工作空间
5. **标识可追溯**：每次审计结果包含审计 agent 标识、模型 ID、时间戳

## 常见错误

**只在高影响时审计。** 中影响变更是双代理互审日常拦截的最大量级——高影响变更本身罕见，"只在高风险时做外部审计"意味着 80% 以上本该被审计的变更被跳过。中影响是对"影响不够大但执行者仍可能犯错"的兜底。

**外部审计 agent 宕了就跳过。** 外部审计 agent 不可用时的正确做法是：高影响 → 阻塞等待（或 break-glass），中影响 → 降级审查 + 24h 补审。无论哪种都不应该无声跳过。

**把降级审查当常态。** 降级审查是用来应对外部审计 agent 临时不可用的应急机制，不是功能替代。如果一个团队降级审查的使用率超过 20%，说明外部审计 agent 的可用性或可靠性需要修复，而不是接受降级为常规操作。

**审计 agent 和执行 agent 使用同一模型提供商。** 外部审计的核心价值是"不同模型提供商的独立视角"。如果执行 agent 和审计 agent 使用同一提供商（哪怕是不同 tier），它们的盲区有一定概率重叠。全同提供商的外部审计效果应当降级考虑。

**在 R4 之后继续追加轮次。** 超过 4 轮意味着收敛无法在规则框架内完成，继续追加只会陷入辩论循环。R4 未收敛 → 升格到人工决策。
