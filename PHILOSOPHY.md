# 设计哲学

## 这个仓库是什么

一套经过实战验证的 AI 辅助开发流水线方法论，以及基于 Claude Code CLI + Codex 的参考实现。

**核心论点**：AI 编码的错误不是随机的——每一类错误对应流水线的一个漏洞。补上漏洞就是消灭该类错误。

- 重复修复同一个症状的不同表象？流水线缺少诊断步骤。
- 跨层缺陷同时漏过三层？流水线缺少全链路审计步骤。
- 误诊为架构问题花了 3 天改一个 10 分钟的问题？流水线缺少"先测最简单的假设"的强制顺序。

流水线的每一步防一类具体错误。跳一步就漏一类。

**为什么需要流水线**：LLM 会跳过诊断直接修复、会过度设计、会忽略简单根因。流水线不是知识——知识它已经够了。流水线是流程：强制执行人和 AI 都倾向于跳过的步骤。

**为什么需要双代理互审**：执行者不知道自己犯错。同一个模型 provider 的 agent 倾向于同类错误。需要不同模型提供商的独立 agent 做外部审计——不是更聪明的执行者，而是视角不同的观察者。

**三层定位**：

| 级别 | 含义 | 适用于 |
|------|------|--------|
| 核心方法论 | 与工具无关，可直接迁移到任何 AI 编码工具 | pipeline 步骤定义、风险层级、双代理互审协议 |
| 参考工具映射 | 展示如何在特定工具上落地方法论，读者可替换 | gstack/superpowers/Codex 的技能映射、步骤对应命令 |
| 参考实现脚本 | 示意用途，已验证环境数=1，不保证覆盖所有环境 | setup 脚本、健康检查、launchd 模板 |

你可以完整采用这套工具链，也可以只取方法论部分适配到你自己的工具生态。

## 给谁用

在 macOS 上使用 Claude Code CLI + Codex（或其他不同模型提供商的独立 agent）的开发者。需要具备：

- 基本的命令行操作能力
- 理解 AI 编码工具的优势和局限
- 愿意为质量增加一些流程步骤（每步 2-10 分钟）

## 不解决什么

- **不替代安全审计**。流水线包含 SAST 扫描和双代理审查，但不覆盖渗透测试、合规审计等专业安全评估。
- **不保证零 bug**。流水线降低特定类别错误的概率，但不消灭所有错误。
- **不覆盖非 macOS 环境的部署细节**。参考实现脚本仅针对 macOS + launchd 验证，Linux/Windows 部署需自行适配。
- **不承诺所有项目一刀切适用**。流水线步骤数和审查级别应根据风险调整，不是每个改动都需要 12 步全跑。
